# 🌍 Flutter 앱 글로벌 로컬라이징 AI 지시서

> **프로젝트 배경**
> 한국어 기반으로 개발된 Flutter 알파 버전 앱(iOS/Android)을 전 세계 사용자를 대상으로 출시하기 위해 다국어 로컬라이징을 적용한다.

---

## 1. 목표 및 범위

### 목표
- 앱 내 모든 사용자 노출 텍스트를 다국어로 지원
- 기기 언어 설정에 따라 자동으로 언어가 전환되는 경험 제공
- 문화권별 UI/UX 차이를 반영한 자연스러운 현지화

### 지원 대상 언어 (우선순위 순)
| 우선순위 | 언어 | 비고 |
|--------|------|------|
| 1 | 한국어 (ko) | 기준 언어 (기존 유지) |
| 2 | 영어 (en) | 글로벌 기본값 |
| 3 | 일본어 (ja) | 주요 아시아 시장 |
| 4 | 중국어 간체 (zh-CN) | 중국 본토 |
| 5 | 중국어 번체 (zh-TW) | 대만/홍콩 |
| 6 | 스페인어 (es) | 중남미/유럽 |
| 7 | 프랑스어 (fr) | 유럽/아프리카 |
| 8 | 독일어 (de) | 유럽 |

## 2. 기획 요구사항

### 2-1. 언어 자동 감지
- 앱 최초 실행 시 기기의 시스템 언어를 자동으로 감지하여 적용
- 지원하지 않는 언어일 경우 영어(en)를 기본값으로 표시
- 한국어 사용자는 기존과 동일한 경험 유지

### 2-2. 언어 수동 변경
- 앱 내 설정 화면에서 사용자가 직접 언어를 변경할 수 있는 기능 제공
- 변경 즉시 앱 전체 언어가 재시작 없이 반영
- 선택한 언어 설정은 앱 종료 후에도 유지 (로컬 저장)

### 2-3. 텍스트 현지화 범위
- 앱 내 모든 화면의 버튼, 레이블, 안내 문구, 오류 메시지 포함
- 푸시 알림 메시지 (서버 연동 시 언어 코드 전달 필요)
- 앱스토어 / 플레이스토어 소개 문구 및 스크린샷 텍스트
- 온보딩, 튜토리얼 화면 포함

## 3. 기능 요구사항

### 3-1. 다국어 텍스트 관리 체계
- 언어별 텍스트 리소스를 중앙에서 일괄 관리할 수 있는 구조로 구성
- 한국어를 기준(템플릿) 언어로 설정하고, 나머지 언어는 동일한 키 구조를 따름
- 번역 누락 키가 발생할 경우 영어 텍스트로 자동 폴백(fallback) 처리
- 향후 언어 추가 시 기존 코드 수정 없이 리소스 파일만 추가하면 적용 가능한 확장 구조

### 3-2. 날짜 / 시간 / 숫자 형식
- 날짜 표기: 언어/지역별 형식 자동 적용
  - 한국: 2026년 5월 11일
  - 미국: May 11, 2026
  - 독일: 11. Mai 2026
- 시간 표기: 12시간제(영어권) / 24시간제(유럽, 한국 등) 자동 구분
- 숫자 및 통화: 천 단위 구분자, 소수점 표기 방식 지역별 적용
  - 예) 1,000.00 (영어권) vs 1.000,00 (독일어권)

### 3-3. RTL(오른쪽→왼쪽) 언어 대응
- 아랍어 등 RTL 언어 선택 시 전체 UI 레이아웃 방향 자동 전환
- 아이콘 방향, 텍스트 정렬, 스크롤 방향 등 RTL 환경에 맞게 반전
- RTL 전용 UI 예외 처리가 필요한 화면 목록 사전 식별 및 명세화

### 3-4. 문화권별 콘텐츠 검토
- 앱 내 이미지, 아이콘, 색상 중 특정 문화권에서 부정적 의미를 가질 수 있는 요소 검토
- 특정 국가에서 법적/문화적으로 민감한 기능은 해당 지역에서 비활성화 처리 가능하도록 설계
- 이모지 및 심볼 사용 시 문화권 중립적 요소 우선 사용

### 3-5. 폰트 다국어 지원
- 한중일(CJK) 문자, 아랍 문자, 라틴 문자 등 모든 지원 언어를 올바르게 렌더링하는 폰트 적용
- 언어별 가독성을 고려한 폰트 사이즈 및 줄 간격 조정 가능 구조

## 4. 번역 프로세스 기획

### 4-1. 번역 작업 흐름
```
한국어 텍스트 정의
    ↓
AI 초벌 번역 (DeepL / ChatGPT 활용)
    ↓
원어민 검수 (주요 언어: 영어, 일본어, 중국어 우선)
    ↓
번역 리소스 파일 등록
    ↓
QA 테스트 (언어별 UI 깨짐, 누락 확인)
    ↓
배포
```

### 4-2. 번역 협업 도구
- 번역 관리 플랫폼(Lokalise 또는 Crowdin 등) 도입 검토
- 번역가, 개발자, 기획자가 동일한 플랫폼에서 협업
- 번역 완료율, 미번역 키 현황을 대시보드로 관리

### 4-3. 번역 품질 기준
- 직역보다 현지 사용자에게 자연스러운 의역 우선
- 앱 내 전문 용어는 용어집(Glossary)을 별도 관리하여 일관성 유지
- 버튼 텍스트는 언어별 최대 글자 수 제한 가이드 제공 (UI 레이아웃 깨짐 방지)

---

## 5. 플랫폼별 현지화 요구사항

### 5-1. iOS
- 지원 언어 목록을 Info.plist에 명시하여 앱스토어 언어 배지 노출
- 시스템 권한 요청 문구(카메라, 위치 등) 다국어 작성 필수
- 앱스토어 커넥트에 언어별 앱 이름, 설명, 키워드, 스크린샷 등록

### 5-2. Android
- 언어별 리소스 폴더 구조에 따라 문자열 리소스 분리 관리
- 구글 플레이 콘솔에 언어별 앱 설명, 그래픽 소재 등록
- 앱 이름(App Label)도 언어별로 현지화 적용

## 6. QA 및 테스트 계획

### 테스트 항목
| 항목 | 내용 |
|------|------|
| 언어 자동 감지 | 각 언어 기기 설정 시 올바른 언어 표시 확인 |
| 폴백 처리 | 미지원 언어 설정 시 영어로 정상 표시 확인 |
| 텍스트 오버플로우 | 긴 번역 텍스트로 인한 UI 레이아웃 깨짐 확인 |
| RTL 레이아웃 | 아랍어 설정 시 전체 UI 방향 전환 확인 |
| 날짜/숫자 형식 | 언어별 올바른 형식 표시 확인 |
| 설정 저장 | 언어 변경 후 앱 재시작 시 설정 유지 확인 |
| 번역 누락 | 모든 화면 텍스트 번역 누락 여부 확인 |

## 8. 향후 확장 고려사항

- 신규 언어 추가 시 번역 리소스 파일만 추가하면 자동 반영되는 구조 유지
- 서버 푸시 알림, 이메일 등 서버 발송 콘텐츠도 사용자 언어 설정 연동
- 지역별 법적 요구사항(개인정보처리방침, 이용약관 등) 언어별 별도 관리
- A/B 테스트 시 언어별 독립적인 실험 집단 구성 고려
