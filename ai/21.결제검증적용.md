# IAP 트랜잭션 로컬 보안 처리 지시서

## 개요

Flutter 인앱 결제(IAP) 구현 시, 서버 없이 로컬에서 트랜잭션을 안전하게 영속화하고 조작을 방지하는 구조를 구현한다.
결제가 필요한 기능 버튼 클릭 시, 미사용 트랜잭션 존재 여부를 먼저 확인하여 불필요한 구매 안내를 생략하고 기능을 즉시 실행한다.

---

## 기술 스택

- **상태관리**: Riverpod (`StateNotifier`)
- **로컬 DB**: SQLite (`sqflite`)
- **보안 저장소**: `flutter_secure_storage`
- **서명**: `crypto` (HMAC-SHA256)
- **루팅 탐지**: `flutter_jailbreak_detection`

---

## 핵심 원칙

1. **DB는 캐시, SecureStorage가 진실의 원천**
   - 트랜잭션의 유효성은 반드시 SecureStorage 서명과 교차 검증한다.
   - DB 단독 조회 결과는 절대 신뢰하지 않는다.

2. **저장과 소비는 DB + SecureStorage 동시 처리**
   - 둘 중 하나만 업데이트하는 코드는 작성하지 않는다.

3. **루팅/탈옥 기기에서는 결제 기능 비활성화**
   - `_init()` 최초 진입 시 반드시 탐지 후 차단한다.

4. **버튼 클릭 시 미사용 트랜잭션 우선 확인**
   - 구매 여부를 먼저 확인하고, 미사용 트랜잭션이 있으면 구매 안내 없이 기능을 즉시 실행한다.
   - 구매 안내(결제 UI)는 미사용 트랜잭션이 없을 때만 표시한다.

## DB 스키마

`purchase_transactions` 테이블에 아래 컬럼을 정의한다.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | TEXT PK | 구매 ID |
| `product_id` | TEXT | 상품 ID |
| `purchased_at` | TEXT | 구매 시각 (ISO8601) |
| `status` | TEXT | `pending` / `consumed` |
| `signature` | TEXT | HMAC-SHA256 서명값 |

> 기존 DB에 추가하는 경우 `onUpgrade` 마이그레이션으로 처리한다.

## 구현 파일 목록

### 1. `PurchaseSecureStorage`

역할: HMAC 서명 생성/저장/조회 및 consumed 상태 관리

구현 사항:
- HMAC-SHA256 서명은 `id | product_id | purchased_at` 조합으로 생성
- 서명 키는 앱 내 상수로 정의 (난독화 권장)
- SecureStorage Android 옵션: `encryptedSharedPreferences: true`
- SecureStorage iOS 옵션: `accessibility: first_unlock`
- 제공 메서드: `saveSignature`, `readSignature`, `markConsumed`, `isConsumed`

### 2. `PurchaseTransactionDao`

역할: DB CRUD + 3중 검증 로직

구현 사항:
- `insert`: DB 저장 시 서명 컬럼 포함, SecureStorage에도 동시 저장
  - `conflictAlgorithm: ConflictAlgorithm.ignore` 로 중복 방지
- `loadPending`: 아래 3단계 검증을 모두 통과한 레코드만 반환
  1. DB 내 `signature` 컬럼 재계산 일치 여부
  2. SecureStorage 서명 교차 일치 여부
  3. SecureStorage `consumed` 상태 확인 → consumed이면 DB도 강제 업데이트 후 제외
- `markConsumed`: DB status 업데이트 + SecureStorage `markConsumed` 동시 호출

### 3. `PurchaseNotifier` 수정

역할: 기존 Notifier에 보안 로직 통합

수정 사항:
- `_init()` 시작 시 루팅/탈옥 탐지 → 탐지되면 `IapStatus.notAvailable` 처리 후 즉시 반환
- `_init()` 에서 상품 로드 전 `dao.loadPending()` 호출하여 복원된 트랜잭션을 state에 주입
- `_handlePurchaseUpdate()` 에서 `PurchaseStatus.purchased` 처리 시 `dao.insert()` 호출
- `resetUnlock()` 에서 consumed 처리할 트랜잭션 ID를 추출하여 `dao.markConsumed()` 호출
- `consumeAdditionalPetCredit()` 동일하게 `dao.markConsumed()` 호출
- 위 두 메서드는 DB 호출이 필요하므로 `async`로 변경

### 4. Provider 수정

- `PurchaseSecureStorage` 인스턴스를 생성하여 `PurchaseTransactionDao`에 주입
- `PurchaseTransactionDao`를 `PurchaseService`에 주입
- 기존 DB 프로바이더에서 `Database` 인스턴스를 참조하여 구성

## 버튼 클릭 처리 흐름

결제가 필요한 기능의 버튼 클릭 핸들러는 아래 흐름을 따른다.
UI 레이어(Widget)에서 직접 분기하며, `PurchaseState`의 getter를 참조한다.

### 데이터 내보내기 버튼

```
버튼 클릭
    │
    ├─ isUnlocked == true  → 구매 안내 없이 내보내기 기능 즉시 실행
    │                         기능 완료 후 resetUnlock() 호출
    │
    └─ isUnlocked == false → 구매 안내 UI 표시 (BottomSheet 또는 Dialog)
                              구매 완료 시 내보내기 기능 실행
                              기능 완료 후 resetUnlock() 호출
```

### 반려동물 추가 버튼

```
버튼 클릭
    │
    ├─ hasAdditionalPetCredit == true  → 구매 안내 없이 반려동물 추가 기능 즉시 실행
    │                                    기능 완료 후 consumeAdditionalPetCredit() 호출
    │
    └─ hasAdditionalPetCredit == false → 구매 안내 UI 표시
                                         buyAdditionalPetCredit() 호출
                                         반환값 true 시 반려동물 추가 기능 실행
                                         기능 완료 후 consumeAdditionalPetCredit() 호출
```

### 구현 위치 및 방식

- 분기 로직은 각 버튼의 `onPressed` 핸들러 또는 해당 화면의 ViewModel/Controller에 작성한다.
- `ref.read(purchaseProvider)` 로 현재 상태를 읽어 `isUnlocked`, `hasAdditionalPetCredit` 를 확인한다.
- 기능 실행 함수와 구매 안내 UI 표시 함수는 별도로 분리하여 각 분기에서 호출한다.
- 구매 안내 UI는 기존 구현을 그대로 활용하되, 미사용 트랜잭션이 없을 때만 호출되도록 한다.

## 보안 검증 흐름

```
loadPending() 호출
    │
    ├─ DB signature 재계산 일치? ──── No → 제외 (DB 조작 탐지)
    │
    ├─ SecureStorage signature 일치? ─ No → 제외 (DB 단독 조작 탐지)
    │
    └─ SecureStorage consumed 상태? ── Yes → DB 강제 consumed 후 제외
                                              (DB 롤백 조작 탐지)
```

## 공격 시나리오별 방어

| 공격 | 방어 |
|---|---|
| DB에 가짜 레코드 INSERT | SecureStorage 서명 없음 → 검증 실패 |
| DB status를 pending으로 롤백 | SecureStorage consumed 상태로 탐지 |
| DB signature 위조 | SecureStorage 교차 검증 불일치 탐지 |
| DB 파일 통째로 교체 | SecureStorage 서명 전무 → 전부 무효 |
| 루팅 후 전체 조작 | 루팅 탐지로 기능 자체 차단 |
